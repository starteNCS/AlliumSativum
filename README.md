# AlliumSativum

Federated Database Management System,

developed as the artifact for the thesis _"Optimizing Distributed Query Execution for Heterogeneous Federated Data Systems via a Novel Cost Model and Domain-Specific Transpilation"_ by Philipp Honsel.

Please make sure to first read the thesis for a better understanding of the system, its design, and its components.

Refer to `Getting started` to try AlliumSativum yourself.

---

## Overview

AlliumSativum is a federated query processing system designed to:
- accept a query request through a central server,
- optimize it using a dedicated planner and cost model,
- execute it across heterogeneous data sources via workers and connectors.

The code is organized into multiple projects, each serving a specific purpose in the overall architecture of the AlliumSativum system.

## Getting started (local development)

This docker compose not only contains the AlliumSativum query server and worker, but also a seeded catalog and three data sources.
These data sources are exactly the same as those used in the thesis evaluation, so this environment allows you to reproduce the evaluation results and experiment with the system on a known dataset.
The queries are already provided in the appendix.

Typical local setup sequence:

1. Start the docker compose environment:
   ```bash
   docker compose -f compose.yml up
   ```
2. Restart the query-server and worker services to ensure they connect to the now-available catalog
3. Execute the `docker-init/init-catalog.sql` script against the catalog database to seed initial metadata (data sources, relations, attributes).
6. Trigger a metrics scrape so the cost model has statistics:
   ```
   GET /metrics/all
   ```
7. Submit a test query:
   ```
   POST /execute
   { "query": "SELECT ..." }
   ```

## End-to-end request flow

1. A client sends a query request to **AlliumSativum.QueryServer**.
2. QueryServer delegates planning/optimization to **AlliumSativum.QueryPlanner**.
3. QueryPlanner produces a **QExP** (query execution plan).
4. QueryServer/Executor passes the QExP to **AlliumSativum.QueryExecutor**.
5. QueryExecutor:
   - executes local plan operators when needed,
   - delegates pushdown fragments to workers.
6. **Workers** choose suitable connectors and run source-specific operations.
7. Results are combined and returned to the client.

## Query language: AsSQL

AlliumSativum uses its own query language called **AsSQL** (AlliumSativum SQL), a federated dialect designed to address heterogeneous sources.

The tokenizer recognises the following token classes via regex:

| Token class    | Examples                        |
|----------------|---------------------------------|
| Literals       | `'hello world'`, 1, 0.94        |
| Operators      | `=`, `!=`, `<`, `>`, `<=`, `>=` |
| Arrow operator | `->`                            |
| Punctuation    | `(`, `)`, `,`, `.`              |
| Identifiers    | table/column names, keywords    |

A typical AsSQL query follows a SQL-like structure with `SELECT`, `FROM`, `JOIN`, and `WHERE` clauses and supports cross-source joins by referencing relations registered in the catalog.

## Query planning pipeline

The `QueryCompiler` in `AlliumSativum.QueryPlanner` orchestrates five sequential stages:

```
Raw AsSQL query string
        │
        ▼
 1. Tokenizer           (AlliumSativum.QueryPlanner / Token)
        │  Stack<string>
        ▼
 2. TokenQueryParser    (AlliumSativum.QueryPlanner / Parser)
        │  SelectDto
        ▼
 3. SemanticTransformer (AlliumSativum.QueryPlanner / Semantic)
        │  Enriched SelectDto
        ▼
 4. Optimizer           (AlliumSativum.QueryPlanner / Optimize)
        │  QueryExecutionPlan(s)
        ▼
 5. Best plan selected  → QueryExecutionPlan (QExP)
```

### Stage details

| Stage | Class | Responsibility                                                                                                                                                                                 |
|-------|-------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Tokenize** | `Tokenizer` | Splits the raw query string into a token stack using a regex lexer.                                                                                                                            |
| **Parse** | `TokenQueryParser` | Builds a `SelectDto` (SELECT / FROM / JOIN / WHERE AST) from the token stack. Implemented as partial classes per clause.                                                                       |
| **Semantic transform** | `SemanticTransformer` | Resolves identifiers against the catalog, applies type information, and validates the model.                                                                                                   |
| **Optimize** | `Optimizer` | Constructs the join tree, pushes down filters and projections, proposes subplans to workers, orders joins by cost, and annotates every node with cardinality, selectivity, and cost estimates. |
| **Select best plan** | `QueryCompiler` | Returns the single cheapest `QueryExecutionPlan` from the optimizer's output if pruning is enabled. Otherwise, returns all possible plans.                                                     |

## Plan operators (POPs)

Every node in a `QueryExecutionPlan` is a `PlanOperator`. Each node carries:

| Property | Description |
|----------|-------------|
| `Cost` | Estimated cost of this node alone. |
| `ExpectedCardinality` | Estimated output row count. |
| `Selectivity` | Fraction of input rows that pass this operator. |
| `Width` | Number of output attributes. |
| `DistributionData` | Per-attribute histogram/distribution metadata. |
| `ExecutionData` | Filled at runtime with actual results and actual cost. |

Concrete operator types:

| Operator                       | Description                                                 |
|--------------------------------|-------------------------------------------------------------|
| `ProjectPlanOperator`          | Column projection / attribute selection.                    |
| `FilterPlanOperator`           | Predicate filtering (on-premises).                          |
| `NestedLoopJoinPlanOperator`   | On-premises nested loop join                                |
| `HashJoinPlanOperator`         | On-premises hash join.                                      |
| `PushdownSqlPlanOperator`      | Pushdown that transpiles to SQL (e.g., PostgreSQL).         |
| `PushdownRestCallPlanOperator` | Pushdown that transpiles to a REST call (e.g., JsonServer). |

## Cost model

The `DefaultCostModel` in `AlliumSativum.Shared/Costs` estimates the cost of every plan operator. All parameters are loaded from configuration (see `CostModelSettings`).

**Total plan cost**
The total cost of a (potentially parallel) plan is the operator's own cost plus the **maximum** child branch cost, reflecting parallel execution of independent sub-plans.

### Configuration keys (`appsettings.json`)

```json
"CostModel": {
  "Project": {
    "BaseCost": ...,
    "PerAttributeCost": ...
  },
  "Filter": {
    "BaseCost": ...,
    "PerAttributeCostNumeric": ...,
    "PerAttributeCostString": ...
  },
  "Join": {
    "NestedLoop": { "BaseCost": ... },
    "Hash": {
      "BaseCost": ...,
      "PerAttributeHashTableInitiation": ...,
      "PerAttributeHashTableLookup": ...,
      "PerPropertyCloneCost": ...
    }
  }
}
```

## QueryServer API endpoints

The QueryServer exposes a minimal HTTP API (no versioning prefix):

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/compile` | Compiles a query and returns the execution plan as pretty-printed HTML. |
| `POST` | `/execute` | Compiles and fully executes a query, returning the result rows. |
| `POST` | `/execute-return-plan` | Compiles and executes a query, returning the annotated plan HTML with total execution time. |
| `GET`  | `/metrics/all` | Triggers a statistics scrape for every registered data source. |
| `GET`  | `/metrics/{datasourceId}` | Triggers a statistics scrape for one specific data source (UUID). |

All `POST` endpoints accept `application/json` with the body:

```json
{ "query": "<AsSQL query string>" }
```

Additional controller endpoints (under `Controllers/`) include:
- **BenchmarkController** – exposes benchmarking utilities for plan quality evaluation.
- **HistogramController** – exposes histogram data used by the cost model.

## Catalog database

AlliumSativum maintains a **catalog** (PostgreSQL) that stores metadata about all registered data sources. The catalog is accessed via `CatalogDatabase` and `DatasourceDatabase`.

Key catalog entities:

| Entity | Table                     | Description |
|--------|---------------------------|-------------|
| `DataSourceEntity` | `catalog.datasources`     | Registered data sources (id, type, connection details). |
| `RelationEntity` | `catalog.relations`       | Tables/endpoints exposed by each source. |
| `AttributeEntity` | `catalog.attributes`      | Columns/fields of each relation with type info. |
| `AttributePeakEntity` | `catalog.attributepeaks` | Cardinality and distribution peaks per attribute, used by the cost model. |

The worker also caches catalog data in **Redis** (`catalog-cache`) to reduce database round-trips during query execution.

## Worker gRPC services

Workers expose three gRPC services defined in `AlliumSativum.Worker.Proto`:

| Service | Responsibility                                                                                                         |
|---------|------------------------------------------------------------------------------------------------------------------------|
| `PlannerService` | Receives a pushdown subplan proposal from the QueryExecutor and returns a (partially) accepted plan.  |
| `ExecutorService` | Receives a finalized pushdown POP and executes it against the data source via the appropriate connector.               |
| `MetricsService` | Triggers a statistics scrape on the worker's connected data source and persists updated histogram data to the catalog. |

Worker-side strategy classes (`Strategies/`) separate planning (`PlannerStrategy`), execution (`ExecutorStrategy`), and statistics collection (`StatisticsStrategy`) concerns.

## Connectors

### PostgreSQL connector (`AlliumSativum.Connectors.PostgreSQL`)

- Transpiles `SelectDto` subtrees into standard SQL.
- Executes the generated SQL via a `DatasourceDatabase` connection.
- Collects column statistics (histograms, cardinalities) during metrics scrapes.

### JsonServer connector (`AlliumSativum.Connectors.JsonServer`)

- Transpiles `SelectDto` subtrees into REST API calls.
- Handles filter/projection pushdown at the HTTP query-parameter level where the JsonServer API allows it.
- Collects statistics by sampling REST responses.

### Shared connector utilities (`AlliumSativum.Connectors.Shared`)

Provides interfaces and helpers shared across connectors:
- `CatalogUtils` – helpers for reading catalog metadata.
- `HttpUtils` – shared HTTP client utilities.
- `Interfaces` – `IConnector` and related abstractions.

## Configuration notes

Configuration is distributed across projects (Aspire.NET should prefill these).

Common things to configure:

| Setting | Location | Description |
|---------|----------|-------------|
| `ConnectionStrings:catalog-database` | `appsettings.json` | PostgreSQL connection string for the catalog. |
| `ConnectionStrings:catalog-cache` | `appsettings.json` (Worker) | Redis connection string for catalog caching. |

| `CostModel` | `appsettings.json` | Cost model parameter block (see cost model section). |
